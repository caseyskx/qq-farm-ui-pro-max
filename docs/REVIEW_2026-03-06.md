# 近期优化操作复盘与影响评估报告 (2026-03-06)

## 1. 微信获取好友机制的底层修复 (SyncAll)
**优化内容**：移除了获取好友时硬编码的平台判断（之前微信走 `GetAll`，QQ走 `SyncAll`）。现在全平台优先尝试 `SyncAll`，失败则自动回退至 `GetAll`。
**带来的影响与评估**：
- **正向影响**：完美解决了部分纯微信登录号无法获取好友列表的死锁缺陷。
- **潜在风险**：如果后续腾讯对微信的 `SyncAll` 接口封禁或限流，系统依靠回退机制仍能存活，保障了双重鲁棒性。但频繁的抛错降级可能会带来额外的几百毫秒网络迟滞。
- **优化建议**：可以考虑在底层加入“降级记忆”缓存，即如果一个账号连续2次触发 `SyncAll` 失败，则在接下来的 24 小时内针对该账号直接首选 `GetAll`，以节约一次必定失败的网络轮询开销。

## 2. 农场计算工具的全面内置节点化 (Node 原生接管)
**优化内容**：将复杂的各类计算功能脱离纯外置的 2800 端口 Python 微服务，改由主 Node 进程内新增的 `farm-calculator.js` 全权接管，并解决了页面暗色模式下的白色残影问题。
**带来的影响与评估**：
- **正向影响**：用户不再需要依赖外部 Python 服务即可使用完整的经验计算、排行和图鉴，显著降低了跨端 Docker 部署与本地初次安装配置的门槛。并节省了系统内存开销。
- **潜在风险**：本地 JavaScript 在处理极大数值或复杂日期运算时，如果不严谨可能会存在与当初 Python 版本微小的结果差异。
- **优化建议**：可在后续补充阶段添加针对 Node 计算引擎的自动化对比对冲测试 (Automated Sync Tests)，确保改写后的算力指标与最初的 Python 模拟版本永远 100% 对齐。

## 3. 日志物理内存阻断机制 (`clearLogs`)
**优化内容**：重装日志数据的分发模型。在账号切换时刻立刻强制全栈执行 `clearLogs()`，阻断了共享池渲染机制。
**带来的影响与评估**：
- **正向影响**：彻底解决了老版本中，当大量挂机且频繁切换多账号时，由于数据未回流导致的上一账号日志闪回渲染现象。
- **潜在风险**：点击切换而接口未拿到新数据前的窗口期，日志面板会产生空洞的视觉留白。
- **优化建议**：除了现在的数据清空，可以在 `clearLogs()` 调用的同时挂载一套优美的占位骨架屏 (Skeleton Loading) 或闪烁条，使视觉连贯性更强。

## 4. 极致并发令牌桶 (Token Bucket) 的重构扩展
**优化内容**：将底层 WebSocket 所有推送队列强行纳入 3 QPS 发送门控，确保最高不触发限流反制。
**带来的影响与评估**：
- **正向影响**：对于高频连发指令（例如极速施肥数百次、批量处理好友列表）被智能削峰垫后，风控触发率降至最低。
- **潜在风险**：普通任务若被强行限流排队过长，可能反噬心跳保活包的按时送达率。虽然我们已加入5次强制出队配额抢占机制，但在最极端状况下依然有失联掉线的潜在微毫风险。
- **优化建议**：可对 Token Bucket 缓冲队列积压量做出警告器侦测，当 Queue Length 异常且长期不下掉时，应主动挂起程序或者直接通知钉钉等外部通道，让管理员提早切除僵死进程。
