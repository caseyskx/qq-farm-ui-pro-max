# 核心功能优化与集成后评估审查报告

**审查时间**：2026-03-07
**对应版本**：v4.5.0
**复查目标**：评估最近（包含微信好友强制兼容、公告管理系统独立化、不偷低级作物过滤等）各项业务功能的集成情况，检视现有架构改动可能带来的潜在风险，并给出相应的优化建议。

---

## 1. 最近功能优化点验证 (Impact Analysis)

### 1.1 微信好友 `force_get_all` 强兼容机制
- **改动面**：对 `friend-actions.js` 下的 `SyncAll` 失效退路进行了增强，在 `Settings.vue` 和 `store.js` 追加了独立的后台开关，并在拦截到 1000020 环境错误时，准许利用老一代 `GetAll` 获取数据。
- **潜在风险与影响**：
  - **正向影响**：完美解决了纯微信号/被封锁号无法获得社交拓扑的问题。
  - **负面风险（已观察）**：`GetAll` 是腾讯封杀较严的流量特征。尽管目前仅作为**备用/强锁模式**开启，但一旦开启，其长期的网络特征可能更容易触碰 `1002003` (频繁限制风控)。
- **优化建议**：在使用 `GetAll` 时，底层的令牌桶 (Token Bucket) 限流应给予 `friend` 层更多的节流让渡（例如在获取包后多抛出一个 `sleep(3000)` 后再进行实质农场访问），尽量抹平其大包特征。

### 1.2 系统公告管理 (Announcement CMS) 独立化与接口调优
- **改动面**：原挂载于 `Settings.vue` 中的公告框已全量下放至全新构建的单路由组件 `AnnouncementManager.vue`。数据使用 MySQL `system_announcements` 持久化，提供“所见即所得”实时预览与带权限开关。同时最近补充了 `syncFromLog` 日志同步功能。
- **潜在风险与影响**：
  - **正向影响**：大大缩减了 `Settings.vue` 逾二千行的庞大业务负担。实现了系统级通知配置解耦，符合后台管理的中台体系。
  - **已修复风险（热修复代码合并遗漏）**：刚才在追踪期间发现因前后端分别迭代，导致 `core/src/controllers/admin.js` 中拉取公告列表引入的 `getAnnouncement` 发生函数名拼写不对齐的问题（应为 `getAnnouncements`）。**现已成功排查并自动修复**，打通了 Vue 侧多列表查询后端的链路。
  - **待优化风险（防雪崩击穿）**：目前的公共数据获取接口 `GET /api/announcement` 若遭受高并发疯狂刷新，有可能击穿数据库。
- **优化建议**：底层的 `database.js` 虽然通过 Redis 增加了 `ANNOUNCEMENT_CACHE_TTL` 缓存，但应当在 Node 进程再套一层防抖 (Debounce) 以阻挡极端突发扫描。

### 1.3 `StealSettings.vue` 底层防亏选项的适配打通
- **改动面**：通过对原始版本的比对，将高级界面中关于“过滤低于白萝卜的农作物”等一系列偷菜防御设定映射到了 `store.js` / `admin.js`。
- **潜在风险与影响**：
  - **无负面影响**：此处代码严格遵循了原有的 Schema 模型与状态拦截规则，UI 与配置库双向同步，规避了旧设置文件无法对接新 API 的风险。

## 2. 遗留观测点与优化建议 (Recommendations)

根据过去阶段的多轮演进复查，以下是需要在近未来关注和防御的点：

1. **路由与权限漏洞防御 (JWT Token 管理)**
   新加入的 `AnnouncementManager.vue` 等路由已注册了 `adminOnly: true`，但在前端通过 `localStorage` 检测很容易在视觉上被绕过（虽然无法拿到存底数据）。**建议**：在 `app.ts` 或 `Menu` 导航卫士层 (`beforeEach`) 加入真正的身份查证路由护城河，防患未然。
   
2. **多终端（WeChat_Pad/Car vs QQ）统一协议兼容上限**
   当前为了在不同端口（Web、车机、iPad 版）扫码通过，利用了第三方的抓取协议和强制代理。我们需要做好第三方 API 一旦抛出 502 等错误时的全局断路降级，防止循环获取验证码卡死前端事件循环。

3. **数据库冗余清洗**
   随着微信 `GetAll` 模式或其它集群大并发执行偷菜动作，日志产出量将成倍增长。建议尽快补全并激活预留的 `logCleanupJob.js`，保证 MySQL 或本地 JSON 的存储冗余量在一个健康的抛物线。
