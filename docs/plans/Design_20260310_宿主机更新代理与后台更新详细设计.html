宿主机更新代理与后台更新详细设计

这份设计在 2026 年 3 月 10 日的仓库状态上继续下钻，目标是把“管理员看到新版本提醒，点击确认后自动更新”拆成可实施的架构，而不直接进入编码。

基线版本：v4.5.17 部署栈：App + MySQL 8 + Redis 7 + ipad860 后台入口：设置页管理员系统区 推荐执行器：宿主机更新代理
版本源
GitHub Releases / Manifest
任务触发
管理员在 Web 后台确认
真正执行
宿主机代理，不在容器内执行
状态承载
MySQL 任务表 + 系统配置缓存

一、最终推荐架构

GitHub Release / latest manifest
-> 应用后台定时检查最新版本
-> system_settings 记录 latest_release 缓存
-> 管理员设置页出现“新版本可更新”提示
-> 管理员点击确认更新
-> 应用后台写入 update_jobs 队列
-> 宿主机更新代理轮询并认领任务
-> 执行 repair-deploy / repair-mysql / update-app
-> 写回任务状态和验证结果
-> 前端轮询看到成功 / 失败 / 回滚结果
核心原则： Web 负责“发现、提醒、授权、展示”；宿主机代理负责“更新、回滚、写日志”。这样可以避免把 Docker 控制权限直接暴露给应用容器。

二、为什么推荐“数据库轮询型宿主机代理”

备选方案 问题 结论
前端按钮直接调 shell 浏览器没有宿主机权限 不可行
应用容器直连 Docker 通常要挂 /var/run/docker.sock，安全边界很差 不推荐
宿主机开放 HTTP 更新接口给容器调用 要解决容器到宿主机连通、鉴权、端口暴露、防误触发 可做，但不是第一期最简方案
宿主机代理轮询 MySQL 中的更新任务 实现简单，权限边界清晰，和当前 Docker 部署结构兼容 第一期推荐

当前标准部署下，MySQL 本来就在 Docker 栈里。应用后台可以轻松写入任务，宿主机代理也可以通过 docker exec 或一次性 MySQL 客户端容器读取和回写任务状态，不需要再引入新的网络控制面。

三、统一安装器继续怎么细化

推荐新增入口

建议新增统一入口脚本，例如 scripts/deploy/install-or-update.sh，它只负责状态识别和流程编排，底层继续复用现有脚本。

建议的参数

参数 含义 默认
--mode auto|install|upgrade|repair 自动识别 / 强制全新安装 / 强制升级 / 进入修复 auto
--upgrade-strategy preserve|overwrite 保留旧目录或覆盖当前目录 preserve
--manifest-url 版本清单地址 GitHub 原始地址
--channel stable|beta 更新通道 stable
--offline-bundle /path 优先从离线包安装 空
--expose-infra-ports 是否暴露 MySQL/Redis/ipad860 到宿主机 关闭
--non-interactive 无人值守部署 关闭

建议额外生成的文件

  • install.meta.json：记录版本、镜像、架构、安装模式、部署时间。
  • install-report.json：记录验证结果和失败原因分类。
  • docker-compose.override.yml：只有在需要暴露 MySQL / Redis / ipad860 端口时才生成。
  • current 与 previous 链接或元数据：用于回滚。

四、更新版本源建议做成统一 Manifest

不要让安装脚本、后台版本检查和发布文档各自猜版本。建议发布时生成一份统一版本清单，例如 latest.json。

字段 示例 用途
version 4.5.18 前后端判断是否有新版本
channel stable 支持稳定版 / 测试版通道
releasedAt 2026-03-12T09:30:00Z 展示时间与排序
appImage smdk000/qq-farm-bot-ui:4.5.18 更新主程序镜像
ipad860Image smdk000/ipad860:20260312 仅当协议服务也要升级时使用
offlineBundleUrl 离线包地址 国内网络和离线部署兜底
sourceArchiveUrl GitHub 源码包 镜像构建兜底
notes 文本或 URL 在后台直接显示更新说明
requiresDbRepair true 提示代理更新前先执行数据库修复
minInstallerVersion 1.0.0 防止旧安装脚本执行新包失败
建议： 第一阶段可以把这份 Manifest 放到 GitHub Release 资产或仓库固定地址。以后如果要换自建更新服务器，只需要替换这份 Manifest 的来源，不必重写 UI 和代理流程。

五、宿主机更新代理详细方案

推荐运行方式

组件 建议 原因
代理脚本 scripts/deploy/update-agent.sh 和现有部署脚本同目录，便于打包与修复
服务托管 systemd service 自动重启、日志可控、适合 Linux 服务器
执行模式 常驻循环或短间隔轮询 第一期实现简单，延迟可接受
互斥锁 flock 或锁文件 确保同一时间只跑一个更新任务

代理职责

  1. 读取部署目录和 .env。
  2. 轮询 update_jobs，认领待执行任务。
  3. 执行预检、备份、部署修复、数据库修复、主程序更新、健康验证。
  4. 把每个阶段的状态和摘要回写到任务表。
  5. 必要时执行回滚。

代理为什么不建议直接写成 Node 服务

当前目标环境未必有宿主机 Node 运行时，但安装 Docker 是必选的。因此第一期代理建议优先采用 shell + Docker 辅助命令的方式。以后如果需要更复杂的并发控制、签名校验或灰度分流，再考虑演进为 Go 或 Node 常驻服务。

六、更新任务表建议

建议新增 update_jobs 表，而不是把所有过程都塞进 system_settings。原因很简单：更新任务天然是日志型、历史型、多次发生的对象。

字段 类型建议 含义
id BIGINT 任务主键
requested_by VARCHAR(100) 触发更新的管理员用户名
current_version VARCHAR(64) 点击确认时的当前版本
target_version VARCHAR(64) 目标版本
channel VARCHAR(32) stable / beta
strategy VARCHAR(32) preserve / overwrite
status VARCHAR(32) 见下面状态机
step VARCHAR(64) 当前执行步骤
progress INT 0-100 的粗粒度进度
agent_host VARCHAR(128) 认领任务的机器
log_path VARCHAR(255) 宿主机日志文件路径
summary TEXT 简短结果说明
details_json JSON 阶段结果、镜像、端口、校验细节
created_at / started_at / finished_at DATETIME 生命周期时间戳

状态建议

queued → claimed → precheck → backup → repair → pull_or_build → deploy → verify → succeeded

失败链路：failed 或 rollback_running → rolled_back / rollback_failed

七、哪些信息放到 system_settings 更合适

键 建议内容 原因
update_config 版本源、Manifest URL、通道、是否允许后台更新、检查周期 是系统级固定配置
update_release_cache 最近一次检查到的最新版本快照 避免每次打开页面都请求 GitHub
update_runtime 最后检查时间、最后成功更新时间、当前活跃任务 ID 快速展示全局状态

这些内容和当前的 global_config 一样，天然属于系统级配置，而不是单次任务历史。

八、后台 API 建议

接口 方法 用途
/api/admin/system-update/overview GET 返回当前版本、最新版本、是否可升级、最近任务摘要
/api/admin/system-update/check POST 管理员手动检查一次新版本
/api/admin/system-update/config GET / POST 读取或保存更新配置
/api/admin/system-update/jobs POST 创建更新任务
/api/admin/system-update/jobs/latest GET 获取最近一条任务状态
/api/admin/system-update/jobs/:id GET 查看某条更新任务详情
/api/admin/system-update/jobs/:id/retry POST 失败后重试

接口权限

  • 只允许超级管理员使用。
  • 写操作必须记录操作人用户名、IP 和时间。
  • 创建更新任务时必须带当前版本和目标版本，后台再做一次比对，防止页面过期误触发。

九、前端页面建议怎么挂

推荐挂点

基于当前代码结构，最合适的入口不是新页面，而是现有 [Settings.vue] 同类管理员系统卡片区域。

位置 建议内容 理由
设置页管理员系统区 “系统更新”卡片 现有第三方 API、自检、集群配置都在这里，用户心智一致
管理员顶部提醒 新版本可更新 badge 让超级管理员一进后台就能看到
通知/公告体系 只同步更新说明，不直接执行更新 普通用户可以看公告，但不应看到更新按钮

为什么更新状态展示建议先用 HTTP 轮询，不先用 Socket

  • 更新过程中应用本身会重启，WebSocket 会断。
  • 轮询更简单，掉线后自动恢复更自然。
  • 当前更新状态变更频率不高，5 秒轮询完全足够。

管理员卡片最少应展示

  • 当前版本
  • 最新版本
  • 发布日期
  • 更新摘要
  • 更新通道
  • 最近检查时间
  • 是否存在进行中的任务
  • “保留旧版本更新” / “覆盖更新” 两种确认按钮

十、管理员提醒逻辑建议

当前数据库里已经有 user_preferences.app_seen_version。这非常适合用来记录“管理员已看过哪个版本的更新提醒”。

条件 行为
当前用户不是管理员 不展示更新按钮，只展示普通公告
管理员且 latest.version > current.version 展示“新版本可更新”提示
管理员已关闭本次提醒 把 app_seen_version 记为最新版本
管理员点击确认更新 创建更新任务，不再只算“已读”
建议： 把“是否已经看过这次更新提醒”与“是否执行过更新”分开。前者用 app_seen_version，后者用 update_jobs。

十一、回滚策略必须提前定清楚

对象 回滚建议 风险说明
部署文件 保留上一版部署目录或备份包 低风险，可回滚
主程序镜像 记录旧镜像 ID 或旧标签，失败时重新起旧镜像 低风险，可回滚
MySQL 结构 第一阶段仅允许加表、加列、加索引这类向前兼容修复 如果将来出现破坏性迁移，必须先全量备份再更新
Redis 通常不需要特殊回滚 主风险较低
ipad860 若单独升级，则保留旧镜像可回滚 ARM 兼容性需先预检

当前仓库里的数据库修复基本是补表、补列、补索引，天然更适合“可回退应用，不强求回退 schema”的模式。这一点在第一期是可接受的，但必须在文档里写明。

十二、端口与手动救援模式的详细建议

推荐做法

  • 默认仍然保持 MySQL / Redis / ipad860 不暴露宿主机端口。
  • 如果用户选择“需要手动外接调试”或“需要远程 DBA/运维协助”，再自动生成 docker-compose.override.yml 暴露端口。
  • 这份 override 文件只在明确需要时生成，避免默认暴露数据库和 Redis。

建议支持的可选变量

变量 默认 说明
MYSQL_HOST_PORT 空 为空表示不暴露 MySQL
REDIS_HOST_PORT 空 为空表示不暴露 Redis
IPAD860_HOST_PORT 空 为空表示不暴露协议服务

十三、安全边界建议

  1. 应用容器不直接拥有 Docker 管理权限。
  2. 更新任务只能由超级管理员发起。
  3. 宿主机代理只执行来自白名单 Manifest 域名和允许镜像仓库的版本。
  4. 若未来支持离线包，必须校验 SHA256。
  5. 更新时必须打锁，禁止并发更新和重复点击。
  6. 所有操作都要审计：谁点的、何时点的、更新到哪个版本、结果如何。

十四、我给你的最终实施建议

优先级 现在该做什么 不要急着做什么
P0 把统一安装器和手动救援模式做完整 不要先做复杂的更新服务器
P1 生成 Manifest、补 update_jobs 表、设计管理员更新卡片 不要先上 WebSocket 更新推送
P2 做宿主机更新代理 + 后台更新按钮 + 轮询状态页 不要给容器直接 Docker 权限
P3 按业务规模决定是否上更新镜像站或专用更新服务器 不要在没有渠道控制需求时过度设计
一句话结论： 你这个需求完全可以先基于 GitHub 做起来，不需要先搭专用更新服务器；但“后台点一下就自动更新”这件事，必须交给宿主机更新代理，而不是交给 Web 容器自己去改自己。

说明：本设计结合当前仓库中的部署脚本、设置页管理员入口、system_settings 全局配置机制、user_preferences 版本字段和现有 MySQL/Redis/ipad860 容器模型整理而成。
