# 根目录瘦身与 Git 主仓切换实施计划

生成日期：2026-03-07

适用目录：`/Users/smdk000/文稿/qq/qq-farm-bot-ui-main_副本`

## 计划目标

本计划解决两个问题：

1. 让当前根目录成为新的 Git 主仓。
2. 在新主仓稳定后，对根目录进行瘦身减负和统一归集。

## 现状结论

- 当前根目录不是 Git 仓库，因为根目录下没有 `.git`。
- 当前真正连接 GitHub 的仓库位于 `github-sync/.git`。
- 旧仓远端为 `https://github.com/smdk000/qq-farm-ui-pro-max.git`。
- 旧仓当前分支为 `main`。
- 旧仓目前存在未提交变更：`github-sync/CHANGELOG.DEVELOPMENT.md`。

## 核心判断

- 可以把 Git 仓库做成当前根目录。
- 可以在保留旧仓历史的前提下，对 Git 仓库做增量更新，并让它切换到当前根目录。
- 可以在新主仓验证通过后，淘汰旧 `github-sync/` 目录。
- 不建议先大规模瘦身，再做 Git 主仓切换。

## 推荐策略

推荐采用“两阶段、四层次”的方案。

### 两阶段

1. 先做 Git 主仓切换。
2. 再做根目录瘦身与统一归集。

### 三层管理模型

1. Git 主仓跟踪层
内容包括：
`core/`、`web/`、`scripts/`、`docker/`、`deploy/`、`nginx/`、根级配置、必要文档、唯一有效服务源码。

2. 本地归档层
内容包括：
`archive/` 下的大文件、历史报告、旧发布物、镜像包、旧副本、原始素材。

3. 运行与缓存层
内容包括：
`data/`、`logs/`、`node_modules/`、`dist/`、`tmp/`。

## 总体实施顺序

1. 先完成主仓切换规划。
2. 再让当前根目录接管旧仓历史。
3. 再单独执行根目录瘦身。
4. 最后正式退役旧 `github-sync/`。

## 阶段 A：主仓切换规划

### 目标

明确以后谁是正式仓库，哪些内容应该进入 Git，哪些内容不能进入 Git。

### 任务

1. 明确继续沿用现有远端仓库。
2. 明确新主仓的 Git 跟踪边界。
3. 明确不进入 Git 的目录和文件。
4. 核对旧仓当前未提交差异。
5. 准备后续归档分区设计。

### 输出物

- Git 跟踪清单
- Git 忽略清单
- 旧仓差异确认单
- 主仓切换方案说明

## 阶段 B：Git 主仓接管

### 目标

让当前根目录成为唯一的 Git 主工作树，并继承旧仓历史。

### 原则

- 不重新开一个断历史的新仓。
- 不把主仓切换和目录瘦身混在一个动作里。
- 第一笔提交只做主仓接管。

### 任务

1. 先做文件级快照或完整备份。
2. 以旧仓历史为源，让当前根目录接管版本控制。
3. 修正 Git 忽略边界。
4. 单独形成“主仓切换基线提交”。
5. 验证新主仓的基础 Git 能力。

### 验收标准

- 当前根目录可以正常执行 Git 基础操作。
- 分支和远端关系清晰。
- 推送链路可用。
- 旧仓历史可追溯。

## 阶段 C：根目录瘦身与统一归集

### 目标

让根目录只保留主工程、部署入口、运行数据入口和统一归档入口。

### 优先级划分

#### P0：高风险先设计，不直接移动

- `data/`
- `core/data/`
- `logs/`
- `scripts/service/logs/`

原因：
这些目录涉及源码运行、Docker 运行和 AI 服务日志，口径目前不统一，不能粗暴合并。

#### P1：优先归集

- `github-sync/`
- `wxcode/`
- `releases/`
- `core/dist/`
- `nc_local_version/`
- `图鉴/`

原因：
这些目录体积大、重复度高、运行期依赖低，适合优先退出根目录主视野。

#### P2：文档归集

- `docs/` 中历史报告
- 预览 HTML
- 审计报告
- 阶段性总结
- 旧 SOP
- 根目录零散历史说明文件

原因：
保留价值高，但不是运行必需内容。

#### P3：最后收尾

- `node_modules/`
- `web/dist/`
- `tmp/`
- `.DS_Store`
- build log

原因：
这些内容可再生，适合在验证通过后再清理。

## 阶段 D：旧仓退役

### 目标

让 `github-sync/` 不再作为主工作目录存在。

### 前提

必须先满足以下条件：

1. 当前根目录已经成为正式 Git 主仓。
2. 新主仓推送验证通过。
3. 本地源码模式验证通过。
4. Docker 部署链路验证通过。
5. 需要保留的旧内容已迁移或归档。

### 退役结果

- `github-sync/` 从主流程中移除。
- 如需保留，可进入 `archive/repo-mirrors/`。
- 如无需保留，可在完成核验后删除。

## 建议保留在新主仓的内容

- `.github/`
- `.gitignore`
- `.dockerignore`
- `.env`
- `README.md`
- `LICENSE`
- `package.json`
- `pnpm-lock.yaml`
- `pnpm-workspace.yaml`
- `docker-compose.yml`
- `core/`
- `web/`
- `scripts/`
- `docker/`
- `deploy/`
- `nginx/`
- `services/` 中唯一有效服务源码
- 少量当前仍然有效的核心文档

## 建议从新主仓剥离或归档的内容

- `archive/` 中大体积归档本体
- `github-sync/`
- `wxcode/`
- `releases/`
- `core/dist/`
- `nc_local_version/`
- `图鉴/`
- AI 工具痕迹目录
- 历史计划、预览和阶段报告
- 运行数据、日志、缓存、依赖、临时文件

## 关键风险

1. 当前根目录不是 Git 仓库。
这意味着在正式接管前，不能依赖 Git 回滚整个工作区。

2. 旧仓存在未提交差异。
如果不先处理，后面容易漏掉旧仓本地修改。

3. 数据和日志路径口径不统一。
这会直接影响后续瘦身动作的安全性。

4. archive 体积过大。
如果错误地整体纳入 Git，会让新主仓重新膨胀。

## 本阶段明确不做的事

- 不执行任何 Git 初始化、切换、提交、推送动作。
- 不移动任何根目录文件或目录。
- 不删除任何旧目录。
- 不修改脚本、配置或代码。
- 不处理运行数据迁移。

## 推荐的最终顺序

1. 先把当前根目录升级为新的 Git 主仓。
2. 再在新主仓内做目录瘦身和统一归集。
3. 最后退役旧 `github-sync/`。
