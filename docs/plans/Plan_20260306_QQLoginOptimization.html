架构设计 码雨云 (QQ农场) API 对接计划

日期: 2026-03-06 | 目标: 优化 QQ 扫码通道，对接 aineishe 第三方授权

📂 1. 业务目标与需求共识 (Consensus)

目标描述

将现有 `qrlogin.js` 中的第三方扫码通道 `qqcode.skeo.net` 替换或优化为新的 `api.aineishe.com` 服务源。需保证 API Token 由“硬编码”向“管理面板动态配置”过渡，增强安全性。本期改造**仅针对 QQ 扫码登录**逻辑。

  • 替换底层底层 `/qqnc/login` 接口调用，放弃旧有的 `skeo.net`。
  • Token 动态化：管理面板支持填入专属 `aineishe_api_token` (默认值设为传入的临时凭据)。
  • 不改变原本前后端组件通讯的主流程 (比如返回图片URL/base64、轮询机制)。

⚠️ 3. 影响面与防御性分析 (Impact Analysis)

[前置输入阻断] UIN 获取问题： 从目前的 API 截图结构看，/qqnc/login 接受的 Query 是 `uin`。通常 QQ 官方扫码在第一步并不需要自己输入账号（扫了才知道是谁）。如果码雨云 API **强制要求输入 uin 才能换取二维码**，那么前端必须修改：在显示二维码前，增加一个“请输入待登录的QQ号”的填写框。
[状态轮询协议破坏] 原 `qqcode.skeo.net` 支持 `/api/qr/check` 来查询状态。码雨云是不是还是调用 `/qqnc/login` 并依靠返回 JSON 来甄别 "等待扫码/成功/过期"？这需要对齐返回的 JSON 结构。

🚀 4. 执行决议准备 (Assess -> Next)

行动指引：由于需要确保不阻断当前架构，我们在用户确认本实施方案后，将按照以下顺序动刀：
