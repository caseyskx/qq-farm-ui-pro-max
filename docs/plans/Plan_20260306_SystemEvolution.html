二、执行路径拆解 (Atomized Milestones)

Phase 1 (P0) 终局化数据库迁移架构

目标： 终结 JSON 及 SQLite 的过渡态，彻底切入 MySQL 主推引擎，解决 account_configs 读写竞争及丢失。

  • 实施脚本迁移，将 data/account_configs.json 数据清洗并灌入 MySQL 的 account_configs 与 accounts 表结构。
  • 更改 data-provider.js 与 store.js，将提取与保存配置的底层驱动置换为 db.query（连接池模式）。
  • 构建进程容错与断线重连守护，当数据库网络瞬断时，不再强制剔除用户授权。
影响面分析 (Impact Analysis):
核心底座变动。重构后，即便短时间内启动百个 Worker，也不会引发 SQLITE_BUSY，配置文件丢失率将降到 0%。初期部署可能涉及老用户的本地 JSON 上云转化，需保证严密的向后兼容。

Phase 2 (P1) 强化队列观测与双列优先仲裁

目标： 解决 TokenBucket 统一 3QPS 引起的“抢收”、“防偷”关键指令被堵塞超时失败的问题。

  • 对底层 UrgentQueue 进行插队逻辑重构（现有的 unshift() 存在倒序和长堵塞风险，需转为正统的双列出栈）。
  • 如果 normalQueue 中的任务排队超过 5 秒，实施超时舍弃或平滑让权策略，确保农作物成熟前 60s 抢收指令必达。
  • 在前端 Dashboard 侧边栏注入「队列积压槽」，实时监测并展现 Worker / Network 的发包拥堵健康度。
影响面分析 (Impact Analysis):
直接决定系统“防偷成功率”。通过将调度“黑盒”透明化，用户可在遇到界面慢指令时立刻直观感受到是“协议在排队”，并能自主决定是否清空队列。

Phase 3 (P1.5)微信环境柔性降级与隔离

目标： 缝合强迫微信账号执行不兼容操作导致的循环报错断联。

  • 引入平台特征策略（Strategy Pattern）。当 config.platform.startsWith('wx') 时，自动摘除高频的全域好友 SyncAll 以及偷菜群发请求。
  • 专设微信长巡周期 (间隔扩大至 15~30 分钟)。
  • 针对微信端的账户卡片添加「受限模式」徽标提醒，引导用户知悉其好友生态功能存在阉割。
影响面分析 (Impact Analysis):
减少不必要的错误探针以及封号牵连，保全主菜地收割功能。降低腾讯侧的风控红线警戒值。

Phase 4 (P2/P3) 内存GC介入与多端告警闭环

目标： 使长线挂机能力突破 72 小时，避免浏览器堆栈溢出，提供第三方预警。

  • 在前端引入 LRU Cache 思路：设置展示栈只保留最近 300 条 worker.logs 及图鉴数据更新，触发 Vue 组件主动垃圾回收 (GC)。
  • 在基础 `pushplus` 外，架设通用 WebHook 配置槽页。发生 P0 级事件（被风控锁定30分钟、账号被踢）时推送飞书/企业微信机器人卡片。
影响面分析 (Impact Analysis):
改善低效能平板（车载浏览器）等终端显示此管理面时的生命周期，实现真正的“7*24h 挂机全自动闭环预警”。
