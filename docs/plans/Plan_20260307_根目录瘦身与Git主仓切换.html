项目根目录瘦身与 Git 主仓切换联合计划

本报告用于回答两个核心问题：一是如何把当前已经膨胀的根目录瘦身并统一归集；二是如何让“当前根目录”接管旧 Git 仓库历史，成为新的唯一主工作目录。

目标 1：当前根目录成为 Git 主仓 目标 2：旧 github-sync 目录退役 本阶段仅输出计划，不执行迁移 根目录当前不是 Git 仓库

当前仓库现状

  • 当前根目录本身没有 .git，因此不是 Git 仓库。
  • 真正连接 GitHub 的仓库位于 github-sync/.git。
  • 旧仓远端为 https://github.com/smdk000/qq-farm-ui-pro-max.git，当前分支为 main。
  • 旧仓当前存在本地未提交变更：github-sync/CHANGELOG.DEVELOPMENT.md。

推荐的总策略

  • 先做“主仓切换”，再做“根目录瘦身”。
  • 不新开一个断历史的新仓，优先继承旧仓历史。
  • 让当前根目录成为唯一主工作树，旧 github-sync/ 退出主流程。
  • 将归档、运行数据、可再生产物从 Git 跟踪边界中剥离。

如何理解“当前根目录不是 Git 仓库”

这句话的准确含义是：当前根目录下没有 Git 元数据目录 .git，因此这个目录里的绝大多数文件虽然真实存在，但并不直接受 Git 管理。你现在看到的根目录其实更像一个“实际开发总工作区”，而旧的 github-sync/ 才是与 GitHub 同步的正式仓库副本。

这并不代表当前根目录不能成为 Git 仓库。相反，它完全可以接管旧仓历史，升级为新的主仓。

是否可以把 Git 仓库做成当前这个目录

可以，而且这是本方案的推荐方向。

但推荐的做法不是“直接重新初始化一个全新 Git 仓库”，而是“继承旧仓历史，让当前根目录成为新的主工作树”。这样做有三个直接收益：

  • GitHub 历史不会断裂，仍然保持连续提交记录。
  • 当前根目录里的真实开发内容能够被正式纳入版本管理。
  • 后续淘汰旧 github-sync/ 目录时，路径关系和协作方式都会更简单。

是否可以给 Git 仓库做增量更新并替换成现在的根目录

问题 结论 说明
能否增量更新 可以 前提是沿用旧仓历史，而不是重新建一个断历史的新仓。
能否替换为当前根目录 可以 推荐把当前根目录升格为唯一主工作区，旧 github-sync/ 退役。
能否淘汰旧 Git 仓库目录 可以 但必须在新主仓验证通过后进行，不能先删旧仓再验证。
是否建议先瘦身再切仓 不建议 先切仓更稳，因为切仓后每一步瘦身都有版本记录和可追踪变更。

推荐的三层管理模型

层级 内容范围 是否进入 Git 管理原则
Git 主仓跟踪层 core/、web/、scripts/、docker/、deploy/、nginx/、根级配置 是 只保留开发、构建、部署、协作必需内容
本地归档层 archive/ 中的大文件、旧报告、历史发布包、镜像包、旧副本、原始素材 原则上否 本地集中保存，保留索引说明，不让仓库持续膨胀
运行与缓存层 data/、logs/、node_modules/、dist/、tmp/ 否 统一忽略规则，避免运行态数据进入仓库

优化后的总体计划

  1. 阶段 A：主仓切换规划。先定义新主仓边界、旧仓接管策略、保留哪些历史、忽略哪些目录。
  2. 阶段 B：Git 主仓接管实施。让当前根目录接管旧仓历史，形成新的唯一工作树。
  3. 阶段 C：目录瘦身与统一归集。在新主仓稳定后，再进行归档、剥离和目录收敛。
  4. 阶段 D：旧仓退役。只有新主仓完成验证后，旧 github-sync/ 才退出主流程。

阶段 A：主仓切换规划

  1. 确认旧仓作为历史源，远端继续沿用现有 GitHub 仓库。
  2. 锁定新主仓的 Git 跟踪清单。
  3. 锁定不进入 Git 的清单，包括归档目录、运行数据、日志、依赖、构建产物。
  4. 核对旧仓当前未提交差异，避免切换时漏掉本地变更。
  5. 为后续根目录瘦身预留清晰的归档分区。
这一阶段的目标不是“整理干净目录”，而是先回答“今后哪个目录才是正式仓库、哪些内容应该被 Git 跟踪”。

阶段 B：Git 主仓接管实施思路

推荐路线是：保留旧仓历史，把当前根目录提升为新的 Git 主工作树，然后将旧 github-sync/ 从“主工作目录”降级为“待退役旧目录”。

  1. 先建立完整快照或文件级备份，原因是当前根目录不是 Git 仓库，不能靠 Git 回退。
  2. 再让当前根目录与旧仓历史建立正式关联。
  3. 第一笔提交只做主仓接管，不混入目录搬迁和归档动作。
  4. 在新主仓上验证 status、分支、远端、推送流程是否正常。
主仓接管和目录瘦身必须拆开。否则一旦出问题，很难判断是仓库切换失败，还是目录归集误伤了启动链路。

阶段 C：根目录瘦身与统一归集

优先级 对象 建议动作 原因
P0 data/、core/data/、logs/、scripts/service/logs/ 先做路径策略设计，暂不移动 这是当前最容易误伤运行链路的高风险区域
P1 github-sync/ 新主仓稳定后退役，并归档或删除 旧主工作目录，不应长期与新主仓并存
P1 wxcode/ 归集到归档区 与 services/ipad860/ 高度重复
P1 releases/、core/dist/ 移入归档区或独立保存区 体积大，可单独管理
P1 nc_local_version/、图鉴/ 归档到原始素材区 运行时已主要使用 web/public/nc_local_version/
P2 docs/ 中历史报告、审计、预览 HTML 归集到 archive/docs/ 有保留价值，但不是运行必需
P2 .cursor/、.agent/、.specstory/、.llm-chat-history/ 归档或剥离出主仓 属于工具痕迹，不属于业务工程本体
P3 node_modules/、web/dist/、tmp/、.DS_Store 验证通过后再清理 属于可再生产物，适合最后收尾

阶段 D：旧仓退役标准

旧 github-sync/ 目录不能“看着差不多就删”，必须满足明确门槛：

  1. 当前根目录已成为正式 Git 主仓。
  2. 新主仓可正常执行版本管理基础操作。
  3. 远端推送验证通过。
  4. 源码模式启动链路验证通过。
  5. Docker 部署链路验证通过。
  6. 归档索引已建立，旧目录需要保留的内容已迁出。

本阶段不做什么

  • 不执行任何 Git 切换操作。
  • 不删除或移动任何根目录文件。
  • 不修改任何启动脚本、Docker 配置或路径代码。
  • 不处理运行数据迁移。
  • 不执行旧仓退役动作。

风险与前置约束

风险点 影响 控制方式
根目录不是 Git 仓库 不能直接用 Git 回滚当前真实工作区 主仓接管前必须先做文件级备份或快照
旧仓有未提交改动 切换时可能遗漏本地历史差异 先核对旧仓差异，再进入主仓切换
数据与日志路径不统一 瘦身时容易误伤启动或运行数据 把路径治理单独作为高风险任务拆开处理
归档目录过大 若直接纳入 Git，会让仓库再次膨胀 将归档本体默认排除在 Git 之外，只保留索引说明

结论

最优顺序不是“先大整理，再搞 Git”，而是：

  1. 先把当前根目录升级为新的 Git 主仓。
  2. 再在新主仓内分批完成根目录瘦身和统一归集。
  3. 最后正式退役旧 github-sync/。
这样能同时满足两个目标：既保住 Git 历史连续性，也能让后续每一轮目录治理都变成可审计、可追踪、可回退的正式工程变更。
