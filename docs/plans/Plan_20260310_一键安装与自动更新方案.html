一键安装与自动更新方案规划

面向当前仓库的部署现状，规划一套可落地的“安装、升级、回退、失败救援、后台提醒更新”完整方案。此阶段只做设计与分解，不编写业务脚本实现。

现有基础
已有安装 / 更新 / 修复脚本骨架
核心缺口
缺统一入口、失败救援流程和后台更新执行链路

一、基于当前仓库的事实判断

现状 仓库依据 规划含义
已经有全新安装脚本 scripts/deploy/fresh-install.sh 可以复用其 Docker 检查、镜像拉取、源码回退构建、健康等待逻辑，不需要从零开始重写。
已经有主程序更新脚本 scripts/deploy/update-app.sh 说明“只更新主程序，不动 MySQL / Redis / ipad860”已经有成熟方向，可作为后台自动更新执行内核。
已经有部署修复和数据库修复 repair-deploy.sh、repair-mysql.sh 后续手动救援模式不必另起炉灶，应该把这两类脚本纳入统一安装器流程。
主程序和 ipad860 已支持“拉镜像失败回源码构建” fresh-install.sh、update-app.sh 你的“主程序镜像拉不下来就去 GitHub 拉源码本地打包”需求，和现有方向一致，但要继续扩成统一决策流。
MySQL / Redis 当前主要依赖官方镜像 deploy/docker-compose.yml MySQL / Redis 没有本仓库源码可本地构建，失败兜底必须走镜像镜像源、离线包或预载镜像，不是 GitHub 回源码构建。
微信扫码服务实际是 ipad860 deploy/docker-compose.yml、services/ipad860/conf/app.conf 它默认内部端口是 8058，依赖 Redis；它并没有像 MySQL 那样的独立用户名 / 密码模型，这一点必须在脚本设计里单独处理。
当前 compose 只对外暴露 Web 端口 deploy/docker-compose.yml MySQL / Redis / ipad860 默认是容器内网访问，安装检查应以内网连通测试为主；如果要外部端口检查，必须新增可选宿主机映射。
系统已有公告与版本感知基础 AnnouncementManager.vue、/api/notifications、user_preferences.app_seen_version 后台“发现新版本提醒超级管理员”可以在现有通知/公告体系上扩，不需要完全新造一套 UI。
关键结论： 当前仓库已经具备 60% 左右的部署基础能力，真正缺的是“统一编排层”和“后台触发更新的执行链路”，不是缺单个 shell 片段。

二、总目标拆解

  1. 做一个统一入口脚本，自动判断是全新安装还是更新。
  2. 安装时自动处理 Docker、镜像拉取、本地构建、配置写入、启动、健康检查、结果汇总。
  3. 更新时支持“保留旧版本目录”和“覆盖当前目录”两种模式，并能回滚。
  4. 如果自动安装失败，切换到手动配置救援模式，逐项引导用户填写 MySQL、Redis、ipad860 相关信息并连通测试。
  5. 把初始化 SQL、修复 SQL、增量迁移脚本、部署修复脚本全部纳入交付包，避免现场安装时还要到处找文件。
  6. 给超级管理员提供“有新版本提示”和“确认后自动更新”的能力。

三、推荐的一键安装器架构

推荐统一入口

建议最终统一成一个宿主机脚本，例如 install-or-update.sh，内部再调度已有的 fresh-install.sh、update-app.sh、repair-mysql.sh、repair-deploy.sh。

统一流程

阶段 核心动作 说明
1. 预检 检测系统、CPU 架构、磁盘、网络、Docker、Compose、端口占用 需要特别检查 ARM 机器是否具备运行 ipad860 的 linux/amd64 兼容能力。
2. 状态识别 判断是否已存在 /opt/qq-farm-bot-current、既有容器、既有部署目录 用来区分全新安装与已有环境升级。
3. 模式选择 全新安装 / 更新安装 / 手动救援 更新安装下再分“保留旧目录安装”和“覆盖当前目录安装”。
4. 资源准备 拉镜像、回退源码构建、准备部署包、准备 SQL 与迁移脚本 主程序与 ipad860 允许源码构建；MySQL / Redis 则走镜像镜像源或离线包。
5. 配置生成 生成或合并 .env，可选映射主机端口，记录安装元数据 这里要把默认值、用户输入值和旧环境继承值分清楚。
6. 启动部署 docker compose up -d，等待容器就绪 启动失败时自动抓取日志，进入救援模式。
7. 安装后验证 验证应用、MySQL、Redis、ipad860 连通性、库表和关键配置 这一步是你要求的“最终确保安装成功不出问题”的关键。
8. 结果展示 输出摘要、写入 install-report.json 和 install-report.html 最终给用户完整展示安装路径、版本、端口、账号、镜像来源、检查结论。

四、全新安装与更新安装怎么定义

全新安装

  • 没有现成部署目录，或者没有正在运行的 qq-farm-bot 相关容器。
  • 安装目录建议继续沿用 /opt/YYYY_MM_DD/qq-farm-bot，并维护 /opt/qq-farm-bot-current 软链。
  • 优先拉取四类镜像：主程序、MySQL 8.x、Redis 7.x、ipad860。

更新安装

这里必须先统一语义，否则脚本后面会很容易设计错。

模式 建议定义 原因
保留原版本 保留旧部署目录与回滚信息，在新目录写入新部署包，再切换 current 链接 这是当前仓库最兼容的做法。旧目录保留，但不建议并行运行两套同名容器。
覆盖安装 备份后清理当前部署目录中的部署文件，再写入最新部署包并更新镜像 适合磁盘紧张、用户明确要覆盖的场景。
并行运行新旧两套 不建议作为第一期默认能力 当前 compose 使用固定容器名、固定数据卷和默认端口，真要并行就必须引入实例 ID、容器名前缀、卷名前缀和端口偏移，复杂度会显著上升。
建议： 第一阶段把“保留原版本”定义成“保留旧目录和回滚能力”，而不是“同时运行两套完整服务”。这样能大幅降低脚本复杂度和后续运维风险。

五、镜像获取与源码回退策略

组件 优先策略 失败后兜底 备注
主程序镜像 直接拉 APP_IMAGE 从 GitHub 拉源码，本地 docker build 按当前平台构建并加载 当前脚本已经有这个方向，建议继续完善构建缓存与失败提示。
ipad860 镜像 直接拉 IPAD860_IMAGE 从 GitHub 拉 services/ipad860 源码本地构建 现有脚本已支持，但 ARM64 机器需要额外兼容性检查。
MySQL 8.x 直接拉官方镜像 改用镜像加速源、私有仓库、离线镜像包、预加载镜像 不能按“GitHub 源码回退构建”思路处理。
Redis 7.x 直接拉官方镜像 改用镜像加速源、私有仓库、离线镜像包、预加载镜像 同上。

推荐增加的交付物

  • 发布时同时产出离线包：主程序镜像包、ipad860 镜像包、MySQL/Redis 官方镜像包、部署文件包。
  • 把 deploy/init-db/01-init.sql、scripts/deploy/*.sh、core/src/database/migrations/*.sql 一并纳入发布包。
  • 优先级顺序建议为：在线拉镜像 > 镜像镜像源 > 本地离线镜像包 > GitHub 源码构建。

六、安装后检查怎么做才算“真正可用”

检查项 建议检查内容 通过条件
主程序 访问 /api/ping，确认返回版本和 ok=true HTTP 正常且版本号可读
MySQL mysqladmin ping、账号密码登录测试、检查核心表是否存在 连接成功且 users、accounts、system_settings、announcements 等关键表存在
Redis redis-cli ping，若启用密码则走鉴权测试 返回 PONG
ipad860 检测容器状态、TCP 8058、HTTP 可访问性、Redis 连接配置是否一致 容器存活，8058 可访问，Redis 口令与主环境一致
配置一致性 校验 .env 与运行容器环境变量 MySQL / Redis / Web / 第三方接口关键项一致
数据库迁移 执行初始化与修复，检查增量迁移是否补齐 没有缺表、缺列、缺索引

关于“端口、用户名、密码”的现实落点

  • MySQL 默认：3306，用户 qq_farm_user，数据库 qq_farm。
  • Redis 默认：6379，密码默认空。
  • ipad860 默认：内部 8058，无独立用户名/密码，主要看端口与 Redis 配置是否一致。
  • 如果你要求安装完后还能从宿主机直接检查这三个服务端口，建议在 compose 中新增可选变量：MYSQL_HOST_PORT、REDIS_HOST_PORT、IPAD860_HOST_PORT。

七、自动安装失败后的“手动救援模式”

这部分必须做成交互式、分步骤，而不是只输出一段报错日志。

救援模块 脚本提示内容 默认值 / 建议值
MySQL 主机、端口、数据库名、用户名、密码、root 密码、是否使用内置容器 mysql / 3306 / qq_farm / qq_farm_user / qq007qq008
Redis 主机、端口、密码、是否使用内置容器 redis / 6379 / 空密码
ipad860 服务地址、端口、是否使用内置容器、Redis 依赖是否共用 ipad860 / 8058
外部微信接口 若不用内置协议服务，则填写 WX_API_URL、WX_API_KEY、WX_APP_ID 默认留空，按外部服务商文档填写

救援模式必须具备的动作

  • 每改一项配置，就立刻做一次单项连通测试，不要让用户全填完再统一失败。
  • 把失败原因分类显示：认证失败、端口不通、服务未启动、库表缺失、协议服务不可访问。
  • 自动拉取缺失的部署文件：01-init.sql、repair-mysql.sh、repair-deploy.sh、update-app.sh。
  • 把增量迁移脚本也预打包进去，不建议现场依赖 GitHub 临时下载。
  • 最终生成一份 connectivity-report.json 和 manual-recovery-report.html。

八、后台“发现新版本并一键更新”的设计

先说结论

第一阶段不需要先搭建专门的更新服务器。 可以先以 GitHub Releases / GitHub 仓库中的版本清单 作为版本来源。 但要实现“后台点确认后自动更新”，仍然必须有宿主机侧的更新执行者，不能只靠 Web 前端按钮本身。

为什么不能只靠 Web 页面按钮

  • Web 页面只是在应用容器里运行，真正要更新的是宿主机上的 Docker 容器和部署目录。
  • 如果让主程序容器直接拿宿主机权限去改自己，通常要挂载 /var/run/docker.sock，安全风险非常高。
  • 更稳妥的做法是：宿主机有一个本地更新代理，Web 后台只发起授权更新任务。

推荐结构

角色 职责 建议实现
版本源 告诉系统“最新版本是多少、更新说明是什么、镜像地址是什么” 先用 GitHub Releases 或仓库中的 latest.json / manifest.json
应用后台 定时检查新版本、提醒管理员、展示更新日志、触发更新请求 复用现有公告、通知、app_seen_version 机制，再补一组更新状态 API
宿主机更新代理 真正执行 update-app.sh、打锁、写日志、回滚 建议是 systemd service 或本地守护进程，而不是直接把 Docker 权限给 Web 容器
前端管理页 只给超级管理员展示“发现新版本”“确认更新”“查看进度”“查看结果” 独立更新卡片或系统设置页中的更新模块

九、GitHub 更新 vs 更新服务器，怎么选

方案 优点 缺点 建议
直接用 GitHub 成本最低，和当前仓库发布方式一致，最快能上线 国内网络不稳定时体验一般；做灰度、渠道包、签名控制能力弱 第一阶段推荐
自建更新服务器 可做灰度发布、私有渠道、镜像中转、国内加速、权限控制、统计 要多一套服务、多一套运维，还要处理可用性与安全 第二阶段按业务量决定
GitHub + 轻量镜像清单镜像站 兼顾上线速度和国内可用性 仍需维护一份镜像清单或静态 CDN 文件 中长期最平衡

我的建议

  • 第一期：版本源用 GitHub，更新执行靠宿主机代理，不先建复杂更新服务器。
  • 第二期：如果面向大量国内用户，再补一个轻量更新清单镜像站或对象存储 CDN。
  • 第三期：只有当你要做商业化、渠道化、灰度控制、签名校验时，再上完整更新服务器。

十、后台自动更新脚本应该怎么设计

建议的执行链路

  1. 后台定时检查版本源，发现 latest_version > current_version。
  2. 把最新版本信息写入 system_settings，并向管理员界面推送新版本提醒。
  3. 管理员点击“确认更新”。
  4. 应用后台创建一条更新任务，发给宿主机更新代理。
  5. 更新代理加锁，防止重复更新。
  6. 更新代理先备份部署信息和当前版本元数据。
  7. 执行 repair-deploy.sh、repair-mysql.sh、update-app.sh，必要时切到指定镜像。
  8. 更新完成后做健康检查，失败则自动回滚并上报失败详情。
  9. 应用后台轮询或通过 WebSocket 获取更新状态，前端展示进度与结果。

脚本层必须补的保护机制

  • 更新锁：同一时间只允许一个更新任务运行。
  • 维护模式：更新期间提示管理员“正在更新，请勿重复操作”。
  • 超时控制：防止拉镜像或构建卡死。
  • 自动回滚：新版本启动失败时，恢复旧目录软链、旧镜像或旧部署文件。
  • 状态持久化：把开始时间、结束时间、执行人、目标版本、结果、失败原因写入数据库或 JSON 任务日志。

前端层建议

  • 只有超级管理员可见“新版本可更新”提示。
  • 普通用户只看发布公告，不看更新按钮。
  • 提示内容建议同时显示：当前版本、最新版本、发布日期、更新摘要、是否建议立即升级。
  • 更新完成后，使用现有公告/通知机制提示“升级成功”或“升级失败”。

十一、建议新增的数据与接口

类型 建议项 用途
system_settings update_source、latest_release、update_policy、last_update_check_at 记录版本源和最近检查结果
更新任务 update_jobs 表或任务 JSON 记录谁触发、更新到哪个版本、是否成功、日志路径
API /api/admin/update/status、/api/admin/update/check、/api/admin/update/confirm 后台查询、手动检查、发起更新
通知 沿用 announcements + app_seen_version 提示管理员“看过这次更新提醒没有”

十二、建议实施顺序

阶段 目标 产出
P0 统一安装/升级入口，打通全新安装、更新安装、失败救援 统一入口脚本、安装报告、人工救援模式、可选宿主机端口映射
P1 补齐离线包、镜像回退、迁移脚本打包与完整验证 离线发布包、迁移包、校验脚本、回滚说明
P2 做后台版本提醒和自动更新执行链路 版本检查 API、更新状态页、宿主机更新代理、更新日志
P3 视业务需要补更新镜像站或专用更新服务器 轻量清单镜像站或完整更新服务

十三、我对这个方案的明确建议

  1. 先做 Linux 首发版。 当前自动装 Docker 的逻辑天然更适合 Linux；Windows/macOS 建议后续单独做 Docker Desktop 引导流程。
  2. 先做“保留旧目录、可回滚”，不要第一期就做并行双实例。
  3. 先用 GitHub 作为版本源。 不要一开始就为了更新提醒去搭一整套更新服务器。
  4. 后台自动更新一定要引入宿主机更新代理。 不建议给 Web 容器直接挂 Docker 管理权限。
  5. 把安装验证报告和手动救援报告做成文件产物。 这样用户、运维、你自己后续排查都会轻松很多。

说明：本报告基于当前仓库中已有的部署脚本、Docker Compose、MySQL 初始化脚本、公告/通知能力和版本字段进行规划；本阶段未对业务代码或部署脚本做功能实现。
