集群滚动更新与账号迁移补充方案

这个补充方案专门回答一个问题：现有分布式能力，能不能在更新时把账号先切到子系统继续运行，主系统更新完再切回。结论是：可以把它优化进更新计划，但当前阶段应该定义成滚动更新 + 排空切换，而不是直接定义成运行中会话无缝热迁移。

结论：可纳入计划 当前能力：近似不停机 暂不承诺：绝对零掉线 新增阶段：P1.5 / P2
现状定位
分布式调度，不是热迁移
可先落地
排空 + 滚动更新
依赖前提
至少双节点执行面
目标效果
大幅缩短中断窗口

一、直接结论

可以优化进更新计划。但要把需求拆成两个层级：

  1. 第一层：利用现有集群能力做 Worker 排空、账号重分配、节点逐台更新，把升级过程变成滚动更新。
  2. 第二层：如果你要求“账号完全无感、不掉线、不重连”，那属于真正的热迁移，当前代码还不具备。
这两个层级不能混在一起讲。现有代码已经能支撑滚动更新思路，但还没有实现“把运行中账号的全部会话态和任务态，从一个进程无缝搬到另一个进程”的机制。

二、为什么说现有分布式能力能帮上忙

代码位置 已具备的能力 对更新的价值
core/src/cluster/master-dispatcher.js Master 维护 accountToWorker 路由表，可对运行账号重新分配并下发到各 Worker 可以把某个节点上的账号重新调度到其他 Worker，适合做滚动更新前的账号迁移
core/src/cluster/worker-client.js Worker 能接收 master:assign:accounts，并按差分执行新增、移除、重启 具备“接管一批账号”和“释放一批账号”的基础执行能力
core/src/runtime/worker-manager.js 账号运行时在独立 Worker 进程/线程内启动、停止、重启 说明账号执行面已经被进程化，天然适合做节点级别的分批切换
core/client.js 支持 FARM_WORKER=1 的 Worker 角色模式，Worker 会连接 Master 领取任务 说明架构上支持控制面与执行面拆开，不必所有东西都绑在一台上更新

三、为什么它现在还不等于“零掉线热迁移”

当前实现更像分布式调度，不是传统意义上的连接级热迁移。要做到你说的“主系统更新时账号不断线，回切也不断线”，至少还缺下面这些关键能力：

  • 运行态导出/导入：当前账号的 socket 连接、登录态、任务队列、定时器状态都在本地 Worker 进程内，没有完整快照迁移协议。
  • 安全切点：没有“等当前动作结束再迁移”的标准 safe point，也没有 draining 状态机。
  • 租约切换：缺少账号级别的分布式租约和抢占保护，无法确保切换时绝不会双跑或抢跑。
  • 双栈切换编排：现有部署脚本还是单套安装栈思路，没有蓝绿或金丝雀更新编排。
所以正确表述应该是：目前能做近似不停机更新，但不能直接承诺所有账号在升级中绝对无感、完全零断线。

四、目前最现实的优化方向

建议把你说的“新建一个子系统接管账号”落成更工程化的方案：备用 Worker 池 / 子集群接管。第一阶段不做热迁移，只做滚动更新。

方案 是否现实 说明
只更新 Master / Web / API，Worker 继续跑 可以优先实现 Worker 与 Master 断链后当前账号仍可能继续执行，但前提是执行面和控制面已拆开部署
把待更新节点先排空，再把账号切给其他 Worker 适合纳入近期计划 这是最合理的第一目标，能把影响压到很小
升级中完全零掉线、会话不重建 当前不现实 需要新增运行态迁移、租约切换、任务恢复等底层能力

五、建议加进更新计划的新阶段

P1.5：集群滚动更新

  • 新增节点状态：active、draining、offline、updating
  • Master 不再给 draining 节点分配新账号
  • 管理员可指定某个 Worker 进入排空模式
  • 排空完成后，由宿主机代理更新该 Worker 所在实例
  • 单节点成功后再切下一台，实现滚动更新

P2：子集群接管更新

  • 支持预热一套备用 Worker 池
  • 更新前先把主集群账号按批次转移到备用池
  • 更新完成后再分批切回
  • 期间增加版本检查、节点健康检查、账号数对账

P3：真正热迁移

  • 账号运行态快照与恢复
  • 任务游标与定时器恢复
  • 账号租约转移和防双跑锁
  • 切换中连接不中断或快速接续
建议的产品口径： 第一阶段对外宣称“支持滚动更新、尽量不停机、显著降低账号中断窗口”，不要一开始就宣称“完全零掉线热升级”。

六、要满足这个方案的前提条件

  1. 部署层必须至少有两套执行节点，单机单容器不成立。
  2. 控制面和执行面最好拆开，Master/Web/API 与 Worker 不要绑死在同一重启单元。
  3. 更新代理必须支持“按节点更新”，而不是只会全局覆盖安装。
  4. 要新增节点健康状态、版本状态和账号分配监控。
  5. 要定义回滚路径，更新失败能把该节点切回旧版本并重新纳管。

七、最终建议

你的方向是对的，但实现口径要收严：不要把“分布式”直接理解成“已经具备热迁移和负载均衡更新”。现有代码已经能支撑“先切走账号再更新节点”的思路，这个很值得加进计划，而且应该加。

我建议把更新计划正式调整成两条线：

  1. 主线：先落地统一安装器、更新任务、宿主机代理、管理员更新提醒。
  2. 增强线：新增集群排空与滚动更新能力，先实现“低中断更新”，后续再评估真正热迁移。
