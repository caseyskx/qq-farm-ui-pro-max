系统更新任务表、API 与时序规格

这一版文档把“下一步怎么开发”落到表结构、接口协议、任务状态机和宿主机代理轮询流程上，目的是让实现可以直接按规格推进。

一、先对齐当前仓库约束

约束 当前仓库事实 规格影响
迁移执行方式 mysql-db.js 不是扫目录批量执行，而是按“缺什么补什么”显式调用迁移文件 新增表不仅要加 017-system-update-jobs.sql，还要在 mysql-db.js 里显式加检测逻辑
接口返回风格 管理员路由大多返回 { ok, data } 或 { ok: false, error } 新接口沿用同样风格，不引入新包装格式
管理员系统入口 设置页已有第三方 API、自检、集群配置等管理员卡片 更新卡片直接挂在设置页系统区
公告已读版本 user_preferences.app_seen_version 已存在 可直接用于“管理员是否看过当前版本提醒”
管理员操作日志 已有 admin_operation_logs，但 scope 目前只支持有限值 更新任务主体放 update_jobs；如要写操作审计，可额外扩一个 system_update scope

二、数据库设计

2.1 新增表：update_jobs

建议单表承载任务排队、执行状态、阶段日志摘要和回滚结果。这样前端、后台、宿主机代理都围绕同一对象协作。

CREATE TABLE IF NOT EXISTS `update_jobs` (
  `id` BIGINT NOT NULL AUTO_INCREMENT,
  `job_uid` VARCHAR(64) COLLATE utf8mb4_unicode_ci NOT NULL,
  `requested_by` VARCHAR(100) COLLATE utf8mb4_unicode_ci NOT NULL,
  `request_source` VARCHAR(32) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT 'web_admin',
  `channel` VARCHAR(32) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT 'stable',
  `strategy` VARCHAR(32) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT 'preserve',
  `current_version` VARCHAR(64) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT '',
  `target_version` VARCHAR(64) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT '',
  `current_app_image` VARCHAR(255) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT '',
  `target_app_image` VARCHAR(255) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT '',
  `manifest_url` VARCHAR(512) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT '',
  `status` VARCHAR(32) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT 'queued',
  `current_step` VARCHAR(64) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT 'queued',
  `progress` INT NOT NULL DEFAULT 0,
  `lock_token` VARCHAR(64) COLLATE utf8mb4_unicode_ci DEFAULT NULL,
  `claimed_by_host` VARCHAR(128) COLLATE utf8mb4_unicode_ci DEFAULT NULL,
  `claimed_by_pid` INT DEFAULT NULL,
  `deploy_dir` VARCHAR(255) COLLATE utf8mb4_unicode_ci DEFAULT '',
  `previous_deploy_dir` VARCHAR(255) COLLATE utf8mb4_unicode_ci DEFAULT '',
  `backup_path` VARCHAR(255) COLLATE utf8mb4_unicode_ci DEFAULT '',
  `log_path` VARCHAR(255) COLLATE utf8mb4_unicode_ci DEFAULT '',
  `summary` VARCHAR(500) COLLATE utf8mb4_unicode_ci DEFAULT '',
  `error_code` VARCHAR(64) COLLATE utf8mb4_unicode_ci DEFAULT '',
  `error_message` VARCHAR(1000) COLLATE utf8mb4_unicode_ci DEFAULT '',
  `details_json` JSON DEFAULT NULL,
  `requested_at` DATETIME DEFAULT CURRENT_TIMESTAMP,
  `claimed_at` DATETIME DEFAULT NULL,
  `started_at` DATETIME DEFAULT NULL,
  `finished_at` DATETIME DEFAULT NULL,
  `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP,
  `updated_at` DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_update_jobs_job_uid` (`job_uid`),
  KEY `idx_update_jobs_status_requested` (`status`, `requested_at`),
  KEY `idx_update_jobs_target_version` (`target_version`, `created_at`),
  KEY `idx_update_jobs_requested_by` (`requested_by`, `created_at`),
  KEY `idx_update_jobs_claimed_host` (`claimed_by_host`, `claimed_at`)
) ENGINE = InnoDB DEFAULT CHARSET = utf8mb4 COLLATE = utf8mb4_unicode_ci;

2.2 字段设计说明

字段 为什么需要
job_uid 给前端和代理一个稳定任务标识，不依赖自增 ID 展示
request_source 区分是 Web 后台触发、CLI 触发，还是未来自动计划任务触发
strategy 标记本次是保留旧目录还是覆盖更新
lock_token 宿主机代理认领任务时写入，避免多代理并发抢同一任务
details_json 保存镜像、端口、自检结果、构建来源、回滚结果等结构化数据

三、系统配置键设计

这部分不需要新建表，直接放进现有 system_settings 即可。

key 内容建议
update_config Manifest URL、通道、检查周期、是否允许后台更新、是否允许覆盖更新、默认策略
update_release_cache 最近一次从 GitHub 拉到的最新版本快照
update_runtime 最后检查时间、最后成功更新时间、当前活跃任务 ID、最近错误

五、接口校验规则

接口 关键校验
创建更新任务 必须是管理员；若已有 queued/claimed/precheck/.../rollback_running 任务，则拒绝再建新任务
保存配置 manifestUrl 必须是白名单域名；checkIntervalMinutes 限定范围
手动检查版本 遵循超时和失败缓存，避免连续击穿 GitHub
重试任务 只能对 failed / rollback_failed / rolled_back 的任务发起

六、宿主机代理轮询规格

6.1 代理循环建议

while true:
  1. 读取 update_config，确认后台更新功能是否启用
  2. 查找最早一条 status = 'queued' 的任务
  3. 尝试用 lock_token + claimed_by_host 原子认领
  4. 认领成功后进入执行阶段
  5. 分阶段回写 status/current_step/progress/summary
  6. 成功则写 succeeded
  7. 失败则进入 rollback_running，再写 rolled_back 或 rollback_failed
  8. sleep 5-10s

6.2 认领 SQL 思路

UPDATE update_jobs
SET
  status = 'claimed',
  current_step = 'claim_job',
  progress = 2,
  lock_token = ?,
  claimed_by_host = ?,
  claimed_by_pid = ?,
  claimed_at = NOW()
WHERE id = ?
  AND status = 'queued';

只有影响行数为 1 时，才算认领成功。

6.3 阶段推进建议

阶段 建议进度 动作
precheck 10 检查 Docker、部署目录、磁盘、端口、QEMU 兼容
backup 20 备份部署文件，必要时备份数据库
repair 35 跑 repair-deploy.sh 和 repair-mysql.sh
pull_or_build 55 拉主程序镜像，失败时回源码构建
deploy 75 执行 update-app.sh 或统一入口脚本
verify 90 验证 MySQL / Redis / ipad860 / HTTP
succeeded 100 回写结果，更新 runtime

八、前端设置页交互状态表

条件 按钮 文案
未检查版本 “检查更新” 显示当前版本和最近检查时间为空
无新版本 仅保留“重新检查” 显示“当前已是最新版本”
有新版本且无活动任务 “保留旧版本更新”“覆盖更新” 显示更新摘要和目标版本
活动任务进行中 禁用提交按钮 显示当前步骤、进度、最近摘要
更新成功 “查看详情” 显示新版本已安装完成
更新失败但已回滚 “查看详情”“重试更新” 明确显示“已自动回滚到旧版本”
回滚失败 “查看详情” 标红，提示需要人工介入

8.1 轮询建议

  • 设置页打开后先拉一次 overview。
  • 若存在活动任务，每 5 秒轮询一次 jobs/latest。
  • 若任务进入终态，再回到 30 秒轮询 overview 或停止轮询。

九、完整时序图

管理员前端 -> 后端 API: POST /system-update/check
后端 API -> GitHub Manifest: 拉取 latest.json
后端 API -> system_settings: 更新 update_release_cache
管理员前端 -> 后端 API: POST /system-update/jobs
后端 API -> MySQL: INSERT update_jobs(status=queued)
宿主机 update-agent -> MySQL: SELECT queued job
宿主机 update-agent -> MySQL: UPDATE claimed + lock_token
宿主机 update-agent -> 本地脚本: repair-deploy / repair-mysql / update-app
宿主机 update-agent -> MySQL: 分阶段回写 progress/current_step
管理员前端 -> 后端 API: GET /system-update/jobs/latest (轮询)
后端 API -> 前端: 返回 status/progress/summary
宿主机 update-agent -> MySQL: 最终写 succeeded 或 rolled_back
后端 API -> system_settings: 更新 update_runtime

十、实现注意点

  1. 迁移文件建议直接命名为 017-system-update-jobs.sql，不要再复用旧编号。
  2. mysql-db.js 里要加显式检测：SHOW TABLES LIKE 'update_jobs'，缺失时执行 017-system-update-jobs.sql。
  3. 更新任务详情不能只靠纯文本日志，至少要写 summary 和 details_json。
  4. 第一阶段不建议实现“取消正在执行的更新”，因为 shell 流程中断和回滚边界复杂。
  5. 若要写管理员操作审计，可考虑把 admin-operation-logs.js 的 VALID_SCOPES 扩到 system_update。
  6. 前端提醒关闭继续复用 /api/view-preferences 里的 appSeenVersion，不必再做新字段。
推荐开发顺序： 1. 迁移表和 service 层模型。2. overview/check/config/jobs API。3. 设置页管理员更新卡片。4. 宿主机代理。5. 统一安装器收口。

说明：本规格严格对齐当前仓库中的迁移执行方式、管理员路由返回风格和设置页挂载结构。落地时不建议跳过 service 层，直接在 controller 里写 SQL。
