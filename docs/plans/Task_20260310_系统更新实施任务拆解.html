系统更新实施任务拆解

这份清单把实现工作拆到具体文件、测试和验收项，默认目标是先交付第一阶段：管理员版本提醒、更新任务模型、后台 API、宿主机更新代理和统一安装器骨架。

实现阶段：P0 + P1 不含：专用更新服务器 前端入口：Settings 管理员系统区 执行环境：Linux 宿主机
后端任务
数据表 + service + routes
前端任务
设置页更新卡片 + 轮询状态
脚本任务
update-agent + install-or-update
测试任务
route/service/script smoke

一、推荐实际开发顺序

  1. 先做数据库和 service 层，先把数据模型站稳。
  2. 再做管理员 API，保证前端能拉数据和发起任务。
  3. 然后做设置页更新卡片和轮询状态。
  4. 最后再做宿主机代理和统一安装器入口。
  5. 不要先写前端按钮，再回头补后端和脚本，那样返工最多。
推荐里程碑： 第 1 个可演示节点不是“真更新成功”，而是“后台能看到最新版本、能创建任务、任务能被模拟推进状态”。这样前后端能尽快合并，不必等脚本全部写完。

四、宿主机脚本实施任务

任务 建议文件 动作
统一安装入口 scripts/deploy/install-or-update.sh 统一识别 install / upgrade / repair 模式，调度已有脚本
宿主机更新代理 scripts/deploy/update-agent.sh 轮询任务、认领任务、推进状态、执行回滚
代理 systemd 文件 scripts/deploy/system-update-agent.service 提供 Linux 宿主机托管模板
Manifest 示例 deploy/latest.example.json 统一版本清单口径，方便前后端和脚本联调
部署文档更新 deploy/README.md、deploy/README.cn.md 增加更新代理安装、启动、排障说明

update-agent.sh 第一版建议最少支持

  • 每 5-10 秒轮询一次 queued 任务
  • 原子认领任务
  • 按阶段回写 progress/current_step/summary
  • 调用 repair-deploy.sh、repair-mysql.sh、update-app.sh
  • 失败时执行回滚并回写 rolled_back / rollback_failed

统一安装入口第一版建议最少支持

  • 自动判断当前机器是全新安装还是升级
  • 自动识别保留旧目录或覆盖更新
  • 统一输出安装报告
  • 失败后进入手动救援模式

推荐最先补的测试

  1. admin-system-update-routes.test.js
  2. system-update-jobs.test.js
  3. settings-system-update.test.mjs

原因很简单：这三项先稳住，前后端主链路就不会一边写一边飘。

六、验收清单

阶段 验收条件
P0.1 数据模型 MySQL 初始化后自动补出 update_jobs 表，service 能正常增删改查
P0.2 后台接口 管理员能成功调用 overview/check/create job/latest job 接口；非管理员全部被拒绝
P0.3 设置页展示 管理员在设置页能看到更新卡片；普通用户看不到更新操作按钮
P0.4 任务轮询 任务进入进行中后，前端能轮询显示步骤和进度
P1.1 宿主机代理 代理能认领 queued 任务并推进到 succeeded 或 rolled_back
P1.2 统一安装器 脚本能判断 install/upgrade/repair，输出安装报告并在失败时进入手动救援模式

七、明确的非目标

  • 第一阶段不做专门的更新服务器。
  • 第一阶段不做多实例并行运行的新旧两套完整环境。
  • 第一阶段不做前端实时 WebSocket 推送更新进度。
  • 第一阶段不做“取消执行中的更新”。
  • 第一阶段不做 Windows/macOS 的完整宿主机更新代理。

八、建议你现在怎么推进

  1. 先做后端数据表和 service。
  2. 再做管理员 API 与设置页卡片。
  3. 确认前端轮询和任务展示稳定后，再接宿主机代理。
  4. 统一安装器放在最后收口，因为它依赖前面所有脚本和状态定义已经稳定。
如果下一步转入编码： 最合理的第一批落地文件是 017-system-update-jobs.sql、 mysql-db.js、 system-update-jobs.js、 system-update-routes.js、 feature-wiring.js、 setting.ts、 Settings.vue。

说明：本任务拆解以当前仓库中的管理员路由拆分方式、设置页集中管理方式和已有测试目录结构为依据，目标是让后续编码按模块推进而不是按想法推进。
