QQ Farm Bot UI 主题全局统一方案（Plan）

目标：参考 JeecgBoot 的主题治理策略，建立统一、可扩展、低改动的主题控制框架；不改业务逻辑，仅改 UI 主题层。

版本：v1.0（计划稿） | 生成时间：2026-03-09 | 参考项目：/Users/smdk/Downloads/JeecgBoot-main/jeecgboot-vue3

1. 结论摘要

本次方案采用“Jeecg 式配置中心 + 运行时主题应用器 + 分层设计令牌”模型，保留你现有的 themeMode、colorTheme、ThemeSettingDrawer 与背景联动逻辑，避免大规模重写。

保持业务逻辑不变 保留现有主题 key 逐步替换硬编码颜色 分阶段降低风险

2. JeecgBoot 可复用的主题策略（提炼）

2.1 配置中心化

  • 通过 projectSetting.ts 统一维护主题相关配置（主题色、暗黑、头部、侧栏、布局）。
  • 通过 store（store/modules/app.ts）持久化并集中修改。

2.2 主题应用器分层

  • changeTheme：处理主色谱变更（含亮暗派生色）。
  • updateDarkTheme：只负责暗黑类名与 data-theme 切换。
  • updateHeaderBgColor / updateSidebarBgColor：负责局部域变量更新（header/sider）。

2.3 事件化设置面板

  • 设置抽屉以枚举事件驱动（HandlerEnum + layoutHandler/baseHandler），可追踪、可扩展。
  • 色板采用预设列表：系统主题色、顶栏、菜单栏分离配置。

2.4 初始化一致性

  • initAppConfig.ts 启动时统一恢复配置并一次性应用主题，避免页面各自做“补丁式改色”。

3. 你当前项目的改造原则（严格约束）

  1. 不改任何业务流程、接口协议、状态机、数据结构。
  2. 保留已有主题 key：default/sakura/cyber/elegant/ocean。
  3. 保留已有主题模式：light/dark/auto，保留背景联动能力。
  4. 新增仅限“主题治理层”：token、映射、样式重构、组件颜色替换。
  5. 采用可回滚方案：每阶段可独立验收并回退。

4. 目标架构（新主题框架）

4.1 三层模型

层级职责示例
L1 设计令牌（Design Tokens） 唯一颜色源，定义语义，不绑定组件 --ui-text-1、--ui-surface-1、--ui-border、--ui-brand-600
L2 主题映射（Theme Mapping） 将 colorTheme + mode 映射到 token 值 default+dark、sakura+light 映射表
L3 组件语义类（Component Semantics） 组件只消费语义类，不直接写灰蓝红色号 .btn-primary/.text-muted/.panel

4.2 语义色域（推荐）

  • Brand：主题色（主按钮、激活态、focus ring）
  • Neutral：背景、文字、边框、分割线
  • Semantic：成功/警告/错误/信息（与品牌色解耦）
  • State：hover/active/disabled/focus

5. 兼容现有逻辑的最小改动方案

5.1 保留不动

  • stores/app.ts 中的主题模式切换逻辑（含 auto 日出日落）。
  • 现有主题抽屉交互流程与字段（主题、背景、性能模式）。
  • 当前主题 class 机制（html.dark、html.theme-*）。

5.2 新增/调整

  • 新增“主题治理配置层”（类似 Jeecg 的 projectSetting + handler 思路）。
  • 把当前散落在页面中的灰色/蓝色硬编码，迁移为 token 语义类引用。
  • 将“蓝色兼容补丁”和页面级 !important 大覆盖，改为正式 token。

6. 分阶段执行计划（Plan）

Phase 0 - 基线冻结与规范确认（0.5 天）

  • 冻结当前主题行为截图基线（light/dark + 5个主题）。
  • 确认语义令牌命名规范与对比度基线（正文 >= 4.5:1）。
  • 输出“禁止新增硬编码色类”的 lint 约束草案。

Phase 1 - 主题内核搭建（1 天）

  • 建立统一 token 表（light/dark + 每个 colorTheme 的 brand 映射）。
  • 建立 runtime apply 函数：一次性把 token 注入到 root 变量。
  • 保持现有 store 字段不变，仅增加“主题应用层调用点”。

Phase 2 - 基础组件统一（1~1.5 天）

  • 优先改：BaseButton/BaseInput/BaseSelect/BaseSwitch/Toast。
  • 全部替换为语义类：不再出现 text-gray-* 等硬编码。
  • 验证组件在 5 套主题 + 深浅模式下视觉一致。

Phase 3 - 布局层统一（0.5~1 天）

  • 改 App.vue / DefaultLayout.vue / Sidebar.vue 的颜色硬编码。
  • 将奖杯按钮等“孤岛配色”纳入语义体系（accent/brand）。
  • 保留原有结构和交互，不改功能。

Phase 4 - 页面重点整治（2~3 天）

  • 按污染优先级：BagPanel、FarmTools、Settings、Accounts、Friends。
  • 逐页把业务页颜色改为 token 语义，去除页面级强制覆盖样式。
  • 保留所有业务行为与接口调用不变。

Phase 5 - 收敛与验收（0.5 天）

  • 删除过时兼容补丁（如蓝色类强制映射）并回归测试。
  • 输出“主题使用指南”和“新增页面接入规范”。

7. 重点风险与规避

风险表现规避策略
深色模式可读性下降 粉色/金色主题在暗背景上文字发灰 正文/辅助文分别绑定 text-1/text-2 token；强制对比度校验
局部组件颜色回归 按钮、标签出现旧蓝色 先改基础组件，再改页面，最后删兼容补丁
改动面过大 难追踪、难回滚 按 Phase 分支提交；每阶段只处理一层
业务误伤 功能逻辑异常 仅改 class/token，不碰业务逻辑与 API

8. 验收标准（Definition of Done）

  1. 5 套主题在 light/dark 下主界面、设置页、账号页、好友页视觉统一。
  2. 全项目不再新增硬编码灰蓝红色类（允许语义异常色域少量白名单）。
  3. 基础组件全部使用语义 token，页面只使用语义类。
  4. 保留并通过：主题切换、深浅切换、auto 模式、背景联动、性能模式。
  5. 无业务回归（账号、任务、批量操作、设置保存等功能正常）。

9. 建议的第一轮交付边界

第一轮仅做 Phase 1 + Phase 2 + Phase 3（先把“框架和基础组件”做稳），暂不全面清理所有业务页，避免一次性改动过大。

第二轮再处理高污染业务页面和历史补丁清理。
